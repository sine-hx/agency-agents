# 🚨 运行手册：事件响应

> **模式**: NEXUS-Micro | **周期**: 数分钟至数小时 | **智能体**: 3-8个

---

## 场景描述

生产环境中出现了问题。用户受到影响。响应速度很重要，但正确处理同样重要。本运行手册涵盖从检测到事后复盘的全过程。

## 严重程度分类

| 级别 | 定义 | 示例 | 响应时间 |
|-------|-----------|----------|--------------|
| **P0 — 关键** | 服务完全中断、数据丢失、安全漏洞 | 数据库损坏、DDoS攻击、认证系统故障 | 立即（全员响应）|
| **P1 — 高** | 主要功能损坏、性能严重退化 | 支付处理中断、50%+错误率、10倍延迟 | < 1小时 |
| **P2 — 中** | 次要功能损坏、有变通方案 | 搜索功能不可用、非关键API错误 | < 4小时 |
| **P3 — 低** | 外观问题、轻微不便 | 样式错误、错别字、轻微界面故障 | 下一个冲刺 |

## 按严重程度的响应团队

### P0 — 关键响应团队
| 智能体 | 角色 | 行动 |
|-------|------|--------|
| **Infrastructure Maintainer（基础设施维护器）** | 事件指挥官 | 评估范围，协调响应 |
| **DevOps Automator（运维自动化）** | 部署/回滚 | 如需要执行回滚 |
| **Backend Architect（后端架构师）** | 根因调查 | 诊断系统问题 |
| **Frontend Developer（前端开发）** | 界面端调查 | 诊断客户端问题 |
| **Support Responder（支持响应器）** | 用户沟通 | 状态页面更新，用户通知 |
| **Executive Summary Generator（执行摘要生成器）** | 利益相关者沟通 | 实时执行更新 |

### P1 — 高优先级响应团队
| 智能体 | 角色 |
|-------|------|
| **Infrastructure Maintainer（基础设施维护器）** | 事件指挥官 |
| **DevOps Automator（运维自动化）** | 部署支持 |
| **相关开发智能体** | 修复实现 |
| **Support Responder（支持响应器）** | 用户沟通 |

### P2 — 中优先级响应
| 智能体 | 角色 |
|-------|------|
| **相关开发智能体** | 修复实现 |
| **Evidence Collector（证据收集器）** | 验证修复 |

### P3 — 低优先级响应
| 智能体 | 角色 |
|-------|------|
| **Sprint Prioritizer（冲刺优先级管理器）** | 添加到待办事项 |

## 事件响应流程

### 第一步：检测与分流（0-5分钟）

```
触发：监控告警 / 用户报告 / 智能体检测

Infrastructure Maintainer:
1. 确认告警
2. 评估范围和影响
   - 有多少用户受影响？
   - 哪些服务受影响？
   - 数据是否有风险？
3. 分类严重程度（P0/P1/P2/P3）
4. 激活相应的响应团队
5. 创建事件频道/线程

输出：事件分类 + 响应团队已激活
```

### 第二步：调查（5-30分钟）

```
并行调查：

Infrastructure Maintainer:
├── 检查系统指标（CPU、内存、网络、磁盘）
├── 查看错误日志
├── 检查最近的部署
└── 验证外部依赖

Backend Architect（如果是P0/P1）:
├── 检查数据库健康状态
├── 查看API错误率
├── 检查服务通信
└── 识别故障组件

DevOps Automator:
├── 查看最近的部署历史
├── 检查CI/CD流水线状态
├── 准备回滚（如需要）
└── 验证基础设施状态

输出：根因已确定（或缩小到组件范围）
```

### 第三步：缓解（15-60分钟）

```
决策树：

IF 由最近部署引起：
  → DevOps Automator: 执行回滚
  → Infrastructure Maintainer: 验证恢复
  → Evidence Collector: 确认修复

IF 由基础设施问题引起：
  → Infrastructure Maintainer: 扩容/重启/故障转移
  → DevOps Automator: 支持基础设施变更
  → 验证恢复

IF 由代码bug引起：
  → 相关开发智能体: 实现热修复
  → Evidence Collector: 验证修复
  → DevOps Automator: 部署热修复
  → Infrastructure Maintainer: 监控恢复

IF 由外部依赖引起：
  → Infrastructure Maintainer: 激活降级/缓存
  → Support Responder: 向用户沟通
  → 监控外部恢复

全程：
  → Support Responder: 每15分钟更新状态页面
  → Executive Summary Generator: 简报利益相关者（仅P0）
```

### 第四步：解决验证（修复后）

```
Evidence Collector:
1. 验证修复解决了问题
2. 工作状态的截图证据
3. 确认没有引入新问题

Infrastructure Maintainer:
1. 验证所有指标恢复正常
2. 确认没有级联故障
3. 修复后监控30分钟

API Tester（如果与API相关）:
1. 对受影响端点运行回归测试
2. 验证响应时间正常化
3. 确认错误率回到基线

输出：事件已解决确认
```

### 第五步：事后复盘（48小时内）

```
Workflow Optimizer 主导事后复盘：

1. 时间线重建
   - 问题何时引入？
   - 何时被检测到？
   - 何时被解决？
   - 总用户影响时长

2. 根因分析
   - 什么失败了？
   - 为什么失败？
   - 为什么没有更早发现？
   - 5个为什么分析

3. 影响评估
   - 受影响用户
   - 收入影响
   - 声誉影响
   - 数据影响

4. 预防措施
   - 什么监控可以更早发现？
   - 什么测试可以预防？
   - 需要什么流程变更？
   - 需要什么基础设施变更？

5. 行动项
   - [行动] → [负责人] → [截止日期]
   - [行动] → [负责人] → [截止日期]
   - [行动] → [负责人] → [截止日期]

输出：事后复盘报告 → Sprint Prioritizer将预防任务添加到待办事项
```

## 沟通模板

### 状态页面更新（Support Responder）

```
[时间戳] — [服务名称] 事件

状态：[调查中 / 已识别 / 监控中 / 已解决]
影响：[用户影响描述]
当前行动：[我们正在采取的措施]
下次更新：[预计下次更新时间]
```

### 执行更新（Executive Summary Generator — 仅P0）

```
事件简报 — [时间戳]

情况：[服务][中断/降级]，影响[N用户/%流量]
原因：[已知/调查中] — [如已知则简述]
行动：[正在采取的措施] — 预计完成时间 [时间预估]
影响：[业务影响 — 收入、用户、声誉]
下次更新：[时间戳]
```

## 升级矩阵

| 条件 | 升级给 | 行动 |
|-----------|------------|--------|
| P0在30分钟内未解决 | Studio Producer | 增加资源，供应商升级 |
| P1在2小时内未解决 | Project Shepherd | 资源重新分配 |
| 疑似数据泄露 | Legal Compliance Checker | 法规通知评估 |
| 用户数据受影响 | Legal Compliance Checker + Executive Summary Generator | GDPR/CCPA通知 |
| 收入影响 > $X | Finance Tracker + Studio Producer | 业务影响评估 |
