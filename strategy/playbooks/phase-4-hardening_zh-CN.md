# 🛡️ 阶段 4 操作指南 — 质量保障与加固

> **持续时间**: 3-7 天 | **智能体数量**: 8 | **把关人**: 现实检验员 (唯一权威)

---

## 目标

最终质量关口。现实检验员的默认判定是"需要返工"——你必须用压倒性的证据证明生产就绪。此阶段的存在是因为首次实施通常需要 2-3 轮修订循环，这是健康的。

## 前置条件

- [ ] 阶段 3 轨迹门已通过 (所有任务已 QA)
- [ ] 阶段 3 移交包已接收
- [ ] 所有功能已实施并逐一验证

## 关键心态

> **现实检验员的默认判定是需要返工。**
>
> 这不是悲观主义——这是现实主义。生产就绪需要:
> - 完整用户旅程端到端可用
> - 跨设备一致性 (桌面、平板、移动端)
> - 负载下的性能 (不只是顺利路径)
> - 安全验证 (不只是"我们加了认证")
> - 规范合规 (每个需求，而非大多数)
>
> 首轮获得 B/B+ 评级是正常和预期的。

## 智能体激活序列

### 步骤 1: 证据收集 (第 1-2 天，全部并行)

#### 📸 证据收集器 — 全面视觉证据

```
激活证据收集器，为 [项目] 收集全面系统证据。

所需交付物:
1. 完整截图套件:
   - 桌面 (1920x1080) — 每个页面/视图
   - 平板 (768x1024) — 每个页面/视图
   - 移动端 (375x667) — 每个页面/视图
2. 交互证据:
   - 导航流程 (点击前后)
   - 表单交互 (空白、已填、已提交、错误状态)
   - 模态框/对话框交互
   - 折叠/可展开内容
3. 主题证据:
   - 浅色模式 — 所有页面
   - 深色模式 — 所有页面
   - 系统偏好检测
4. 错误状态证据:
   - 404 页面
   - 表单验证错误
   - 网络错误处理
   - 空状态

格式: 附带 test-results.json 的截图证据包
时间线: 2 天
```

#### 🔌 API 测试器 — 全 API 回归

```
激活 API 测试器，对 [项目] 进行完整 API 回归测试。

所需交付物:
1. 端点回归套件:
   - 所有端点已测试 (GET、POST、PUT、DELETE)
   - 认证/授权验证
   - 输入验证测试
   - 错误响应验证
2. 集成测试:
   - 跨服务通信
   - 数据库操作验证
   - 外部 API 集成
3. 边缘情况测试:
   - 速率限制行为
   - 大载荷处理
   - 并发请求处理
   - 格式错误输入处理

格式: 附带每个端点通过/失败的 API 测试报告
时间线: 2 天
```

#### ⚡ 性能基准测试器 — 负载测试

```
激活性能基准测试器，对 [项目] 进行负载测试。

所需交付物:
1. 10 倍预期流量负载测试:
   - 响应时间分布 (P50、P95、P99)
   - 负载下吞吐量
   - 负载下错误率
   - 资源利用率 (CPU、内存、网络)
2. Core Web Vitals 测量:
   - LCP (最大内容绘制) < 2.5s
   - FID (首次输入延迟) < 100ms
   - CLS (累积布局偏移) < 0.1
3. 数据库性能:
   - 查询执行时间
   - 连接池利用率
   - 索引有效性
4. 压力测试结果:
   - 破坏点识别
   - 优雅降级行为
   - 过载后恢复时间

格式: 性能认证报告
时间线: 2 天
```

#### ⚖️ 法律合规检查器 — 最终合规审计

```
激活法律合规检查器，对 [项目] 进行最终合规审计。

所需交付物:
1. 隐私合规验证:
   - 隐私政策准确性
   - 同意管理功能
   - 数据主体权利实施
   - Cookie 同意实施
2. 安全合规:
   - 数据加密 (静态和传输中)
   - 认证安全
   - 输入净化
   - OWASP Top 10 检查
3. 监管合规:
   - GDPR 要求 (如适用)
   - CCPA 要求 (如适用)
   - 行业特定要求
4. 无障碍合规:
   - WCAG 2.1 AA 验证
   - 屏幕阅读器兼容性
   - 键盘导航

格式: 合规认证报告
时间线: 2 天
```

### 步骤 2: 分析 (第 3-4 天，并行，在步骤 1 之后)

#### 📊 测试结果分析器 — 质量指标聚合

```
激活测试结果分析器，聚合 [项目] 的质量指标。

输入: 所有步骤 1 报告
所需交付物:
1. 综合质量仪表板:
   - 整体质量分数
   - 类别分解 (视觉、功能、性能、安全、合规)
   - 问题严重性分布
   - 趋势分析 (如有多轮测试循环)
2. 问题优先级排序:
   - 关键问题 (生产前必须修复)
   - 高优先级问题 (生产前应修复)
   - 中等问题 (下个冲刺修复)
   - 低优先级问题 (放入待办)
3. 风险评估:
   - 生产就绪概率
   - 残留风险领域
   - 建议缓解措施

格式: 质量指标仪表板
时间线: 1 天
```

#### 🔄 工作流优化器 — 流程效率审查

```
激活工作流优化器，对 [项目] 进行流程效率审查。

输入: 阶段 3 执行数据 + 步骤 1 发现
所需交付物:
1. 流程效率分析:
   - Dev↔QA 循环效率 (首过率、平均重试次数)
   - 瓶颈识别
   - 不同问题类型的解决时间
2. 改进建议:
   - 阶段 6 运营的流程变更
   - 自动化机会
   - 质量改进建议

格式: 优化建议报告
时间线: 1 天
```

#### 🏗️ 基础设施维护者 — 生产就绪检查

```
激活基础设施维护者，检查 [项目] 的生产就绪状态。

所需交付物:
1. 生产环境验证:
   - 所有服务健康且响应正常
   - 自动扩展已配置并测试
   - 负载均衡器配置已验证
   - SSL/TLS 证书有效
2. 监控验证:
   - 所有关键指标正在收集
   - 告警规则已配置并测试
   - 仪表板访问已验证
   - 日志聚合工作正常
3. 灾难恢复验证:
   - 备份系统可用
   - 恢复流程已文档化并测试
   - 故障转移机制已验证
4. 安全验证:
   - 防火墙规则已审查
   - 访问控制已验证
   - 秘密管理已确认
   - 漏洞扫描干净

格式: 基础设施就绪报告
时间线: 1 天
```

### 步骤 3: 最终判定 (第 5-7 天，顺序执行)

#### 🔍 现实检验员 — 最终裁决

```
激活现实检验员，对 [项目] 进行最终集成测试。

强制流程 — 不可跳过:

步骤 1: 现实检验命令
- 验证实际构建的内容 (ls、grep 搜索声称的功能)
- 将声称的功能与规范交叉核对
- 运行综合截图采集
- 审查步骤 1 和步骤 2 的所有证据

步骤 2: QA 交叉验证
- 审查证据收集器发现
- 与 API 测试器结果交叉参考
- 验证性能基准测试器数据
- 确认法律合规检查器发现

步骤 3: 端到端系统验证
- 测试完整用户旅程 (而非单个功能)
- 验证所有设备的响应式行为
- 检查端到端交互流程
- 审查实际性能数据

步骤 4: 规范现实检验
- 引用原始规范的**确切文本**
- 与**实际**实施证据对比
- 文档化规范与现实间的每个差距
- 不做假设 — 仅凭证据

判定选项:
- 就绪: 生产就绪的压倒性证据 (首轮罕见)
- 需要返工: 已识别具体问题，附带修复列表 (预期)
- 未就绪: 需返回阶段 1/2 的重大架构问题

格式: 基于现实的集成报告
默认: 需返工，除非另有证明
```

## 质量门 — 最终门

| # | 标准 | 阈值 | 所需证据 |
|---|------|------|---------|
| 1 | 用户旅程完整 | 所有关键路径端到端可用 | 现实检验员截图 |
| 2 | 跨设备一致性 | 桌面 + 平板 + 移动端均可用 | 响应式截图 |
| 3 | 性能认证 | P95 < 200ms, LCP < 2.5s, 可用性 > 99.9% | 性能基准测试器报告 |
| 4 | 安全验证 | 零关键漏洞 | 安全扫描 + 合规报告 |
| 5 | 合规认证 | 所有监管要求已达成 | 法律合规检查器报告 |
| 6 | 规范合规 | 100% 规范需求已实施 | 点对点验证 |
| 7 | 基础设施就绪 | 生产环境已验证 | 基础设施维护者报告 |

## 门决策

**唯一权威**: 现实检验员

### 如果就绪 (进入阶段 5):

```markdown
## 阶段 4 → 阶段 5 移交包

### 供发布团队:
- 现实检验员认证报告
- 性能认证
- 合规认证
- 基础设施就绪报告
- 已知限制 (如有)

### 供增长黑客:
- 产品已就绪供用户使用
- 营销信息的功能列表
- 证明可信度的性能数据

### 供 DevOps 自动化器:
- 生产部署已批准
- 蓝绿部署计划
- 回滚流程已确认
```

### 如果需要返工 (返回阶段 3):

```markdown
## 阶段 4 → 阶段 3 返回包

### 修复列表 (来自现实检验员):
1. [关键问题 1]: [描述 + 证据 + 修复指令]
2. [关键问题 2]: [描述 + 证据 + 修复指令]
3. [高优先级问题 1]: [描述 + 证据 + 修复指令]
...

### 流程:
- 问题进入 Dev↔QA 循环 (阶段 3 机制)
- 每个修复必须通过证据收集器 QA
- 所有修复完成后 → 返回阶段 4 步骤 3
- 现实检验员用更新证据重新评估

### 预期: 2-3 轮修订循环是正常的
```

### 如果未就绪 (返回阶段 1/2):

```markdown
## 阶段 4 → 阶段 1/2 返回包

### 已识别的架构问题:
1. [根本问题]: [为何无法在阶段 3 修复]
2. [结构性问题]: [架构层面需要改变什么]

### 建议行动:
- [ ] 修订系统架构 (阶段 1)
- [ ] 重建基础 (阶段 2)
- [ ] 削减范围并重新定义 (阶段 1)

### 需制作总监决策
```

---

*阶段 4 在现实检验员发布就绪判定并附带压倒性证据时完成。需要返工是预期的首轮结果——这意味着系统可用但需要打磨。*
