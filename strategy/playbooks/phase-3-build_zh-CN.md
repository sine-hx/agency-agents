# 🔨 阶段 3 操作指南 — 构建与迭代

> **持续时间**: 2-12 周 (因范围而异) | **智能体数量**: 15-30+ | **把关人**: 智能体协调器

---

## 目标

通过持续的 Dev↔QA 循环实施所有功能。每个任务在下一个开始前都要经过验证。这是工作量最大的阶段，也是 NEXUS 协调发挥最大价值的阶段。

## 前置条件

- [ ] 阶段 2 质量门已通过 (基础已验证)
- [ ] 冲刺优先排序器待办事项可用，附带 RICE 评分
- [ ] CI/CD 管道可运行
- [ ] 设计系统和组件库就绪
- [ ] API 脚手架，附带认证系统就绪

## Dev↔QA 循环 — 核心机制

智能体协调器通过此循环管理每个任务:

```
对于冲刺待办中的每个任务 (按 RICE 评分排序):

  1. 将任务分配给适当的开发者智能体 (见分配矩阵)
  2. 开发者实施任务
  3. 证据收集器测试任务
     - 视觉截图 (桌面、平板、移动端)
     - 根据验收标准进行功能验证
     - 品牌一致性检查
  4. 如果判定 == 通过:
       标记任务完成
       进入下一个任务
     否则，如果判定 == 失败 且尝试次数 < 3:
       将 QA 反馈发送给开发者
       开发者修复具体问题
       返回步骤 3
     否则，如果尝试次数 >= 3:
       上报至智能体协调器
       协调器决定: 重新分配、分解、延期或接受
  5. 更新管道状态报告
```

## 并行构建轨道

对于 NEXUS-Full 部署，四条轨道同时运行:

### 轨道 A: 核心产品开发

```
管理者: 智能体协调器 (Dev↔QA 循环)
智能体: 前端开发者、后端架构师、AI 工程师、
        移动应用构建器、高级开发者
QA: 证据收集器、API 测试器、测试结果分析器

冲刺节奏: 2 周冲刺
每日: 任务实施 + QA 验证
冲刺结束: 冲刺评审 + 回顾
```

### 轨道 B: 增长与营销准备

```
管理者: 项目协调者
智能体: 增长黑客、内容创作者、社交媒体策略师、
        应用商店优化器

冲刺节奏: 与轨道 A 里程碑对齐
活动:
- 增长黑客 → 设计病毒传播循环和推荐机制
- 内容创作者 → 构建发布内容管道
- 社交媒体策略师 → 规划跨平台营销活动
- 应用商店优化器 → 准备商店页面 (如为移动端)
```

### 轨道 C: 质量与运营

```
管理者: 智能体协调器
智能体: 证据收集器、API 测试器、性能基准测试器、
        工作流优化器、实验追踪器

持续活动:
- 证据收集器 → 每个任务的截图 QA
- API 测试器 → 每个 API 任务的端点验证
- 性能基准测试器 → 定期负载测试
- 工作流优化器 → 流程改进识别
- 实验追踪器 → 为已验证功能设置 A/B 测试
```

### 轨道 D: 品牌与体验打磨

```
管理者: 品牌守护者
智能体: UI 设计师、品牌守护者、视觉叙事者、
        趣味注入器

触发式活动:
- UI 设计师 → QA 发现视觉问题时进行组件精细化
- 品牌守护者 → 定期品牌一致性审计
- 视觉叙事者 → 功能完成时创建视觉叙事素材
- 趣味注入器 → 微交互和惊喜时刻
```

## 冲刺执行模板

### 冲刺规划 (第 1 天)

```
冲刺优先排序器激活:
1. 查看附带更新 RICE 评分的待办事项
2. 根据团队速度选择冲刺任务
3. 将任务分配给开发者智能体
4. 识别依赖关系和排序
5. 设定冲刺目标和成功标准

输出: 附带任务分配的冲刺计划
```

### 每日执行 (第 2 天至第 N-1 天)

```
智能体协调器管理:
1. 当前任务状态检查
2. Dev↔QA 循环执行
3. 阻塞问题识别和解决
4. 进度跟踪和报告

状态报告格式:
- 今日完成的任务: [列表]
- 正在 QA 的任务: [列表]
- 正在开发的任务: [列表]
- 阻塞的任务: [列表及原因]
- QA 通过率: [X/Y]
```

### 冲刺评审 (第 N 天)

```
项目协调者主持:
1. 演示已完成功能
2. 审查每个任务的 QA 证据
3. 收集利益相关方反馈
4. 根据经验教训更新待办事项

参与者: 所有活跃智能体 + 利益相关方
输出: 冲刺评审摘要
```

### 冲刺回顾

```
工作流优化器主持:
1. 哪些做得好?
2. 哪些可以改进?
3. 下个冲刺我们将改变什么?
4. 流程效率指标

输出: 回顾行动项
```

## 协调器决策逻辑

### 任务失败处理

```
当任务 QA 失败:
  如果尝试次数 == 1:
    → 将具体 QA 反馈发送给开发者
    → 开发者仅修复已识别的问题
    → 重新提交 QA

  如果尝试次数 == 2:
    → 发送累积的 QA 反馈
    → 考虑: 开发者智能体是否适合?
    → 开发者附带额外上下文进行修复
    → 重新提交 QA

  如果尝试次数 == 3:
    → 上报
    → 选项:
      a) 重新分配给不同的开发者智能体
      b) 将任务分解为更小的子任务
      c) 修改方法/架构
      d) 已知限制下接受 (文档化)
      e) 延期至未来冲刺
    → 文档化决策和理由
```

### 并行任务管理

```
当多个任务无依赖关系:
  → 同时分配给不同开发者智能体
  → 每个独立运行 Dev↔QA 循环
  → 协调器同时跟踪所有循环
  → 按依赖顺序合并完成的任务

当任务有依赖关系:
  → 等待依赖通过 QA
  → 然后分配依赖任务
  → 在移交中包含依赖上下文
```

## 门决策

**把关人**: 智能体协调器

- **通过**: 功能完整的应用 → 激活阶段 4
- **继续**: 需要更多冲刺 → 继续阶段 3
- **上报**: 系统性问题 → 制作总监干预
