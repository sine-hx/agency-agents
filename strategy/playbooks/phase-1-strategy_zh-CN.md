# 🏗️ 阶段 1 操作指南 — 战略与架构

> **持续时间**: 5-10 天 | **智能体数量**: 8 | **把关人**: 制作总监 + 现实检验员

---

## 目标

在编写第一行代码之前，定义我们要构建什么、如何构建以及成功标准。每个架构决策都要文档化。每个功能都要优先级排序。每一分钱都要有账可查。

## 前置条件

- [ ] 阶段 0 质量门已通过 (GO 决策)
- [ ] 阶段 0 移交包已接收
- [ ] 利益相关方对项目范围达成共识

## 智能体激活序列

### 步骤 1: 战略框架 (第 1-3 天，并行)

#### 🎬 制作总监 — 战略组合对齐

```
激活制作总监，对 [项目] 进行战略组合对齐。

输入: 阶段 0 执行摘要 + 市场分析报告
所需交付物:
1. 附带项目定位的战略组合计划
2. 愿景、目标和 ROI 目标
3. 资源分配策略
4. 风险/回报评估
5. 成功标准和里程碑定义

对齐: 组织战略目标
格式: 战略组合计划模板
时间线: 3 天
```

#### 🎭 品牌守护者 — 品牌身份系统

```
激活品牌守护者，对 [项目] 进行品牌身份开发。

输入: 阶段 0 UX 研究 (人物画像、旅程地图)
所需交付物:
1. 品牌基础 (宗旨、愿景、使命、价值观、个性)
2. 视觉身份系统 (颜色、字体、间距，以 CSS 变量形式)
3. 品牌调性和信息架构
4. Logo 系统规范 (如为新品牌)
5. 品牌使用指南

格式: 品牌身份系统文档
时间线: 3 天
```

#### 💰 财务追踪器 — 预算与资源规划

```
激活财务追踪器，对 [项目] 进行财务规划。

输入: 制作总监战略计划 + 阶段 0 技术栈评估
所需交付物:
1. 附带类别明细的综合项目预算
2. 资源成本预测 (智能体、基础设施、工具)
3. ROI 模型，附带盈亏平衡分析
4. 现金流时间线
5. 财务风险评估，附带应急储备

格式: 附带 ROI 预测的财务计划
时间线: 2 天
```

### 步骤 2: 技术架构 (第 3-7 天，并行，在步骤 1 输出可用后)

#### 🏛️ UX 架构师 — 技术架构 + UX 基础

```
激活 UX 架构师，对 [项目] 进行技术架构设计。

输入: 品牌守护者视觉身份 + 阶段 0 UX 研究
所需交付物:
1. CSS 设计系统 (变量、令牌、尺度)
2. 布局框架 (Grid/Flexbox 模式、响应式断点)
3. 组件架构 (命名约定、层级结构)
4. 信息架构 (页面流程、内容层级)
5. 主题系统 (浅色/深色/系统跟随切换)
6. 无障碍基础 (WCAG 2.1 AA 基线)

需创建的文件:
- css/design-system.css
- css/layout.css
- css/components.css
- docs/ux-architecture.md

格式: 开发就绪的基础包
时间线: 4 天
```

#### 🏗️ 后端架构师 — 系统架构

```
激活后端架构师，对 [项目] 进行系统架构设计。

输入: 阶段 0 技术栈评估 + 合规要求
所需交付物:
1. 系统架构规范
   - 架构模式 (微服务/单体/无服务器/混合)
   - 通信模式 (REST/GraphQL/gRPC/事件驱动)
   - 数据模式 (CQRS/事件溯源/CRUD)
2. 附带索引策略的数据库 Schema 设计
3. 附带版本管理的 API 设计规范
4. 认证与授权架构
5. 安全架构 (纵深防御)
6. 可扩展性计划 (横向扩展策略)

格式: 系统架构规范
时间线: 4 天
```

#### 🤖 AI 工程师 — ML 架构 (如适用)

```
激活 AI 工程师，对 [项目] 进行 ML 系统架构设计。

输入: 后端架构师系统架构 + 阶段 0 数据审计
所需交付物:
1. ML 系统设计
   - 模型选择和训练策略
   - 数据管道架构
   - 推理策略 (实时/批量/边缘)
2. AI 伦理与安全框架
3. 模型监控和重训练计划
4. 与主应用的集成点
5. ML 基础设施成本预测

条件: 仅在项目包含 AI/ML 功能时激活
格式: ML 系统设计文档
时间线: 3 天
```

#### 👔 高级项目经理 — 规范到任务转换

```
激活高级项目经理，为 [项目] 创建任务列表。

输入: 所有阶段 0 文档 + 架构规范 (可用时)
所需交付物:
1. 综合任务列表
   - 引用规范的**确切需求** (不加豪华功能)
   - 每个任务有明确的验收标准
   - 任务间依赖关系已映射
   - 工作量估算 (故事点或小时)
2. 工作分解结构
3. 关键路径识别
4. 实施风险登记册

规则:
- 不要添加规范中未包含的功能
- 引用需求的确切文本
- 工作量估算要切合实际

格式: 附带验收标准的任务列表
时间线: 3 天
```

### 步骤 3: 优先级排序 (第 7-10 天，顺序执行，在步骤 2 之后)

#### 🎯 冲刺优先排序器 — 功能优先级排序

```
激活冲刺优先排序器，对 [项目] 进行待办事项优先级排序。

输入:
- 高级项目经理 → 任务列表
- 后端架构师 → 系统架构
- UX 架构师 → UX 架构
- 财务追踪器 → 预算框架
- 制作总监 → 战略计划

所需交付物:
1. RICE 评分的待办事项 (触达、影响、置信度、工作量)
2. 冲刺分配，附带基于速度的估算
3. 依赖关系图，附带关键路径
4. MoSCoW 分类 (必须有/应该有/可以有/不会有)
5. 发布计划，附带里程碑映射

验证: 制作总监确认战略对齐
格式: 优先级排序的冲刺计划
时间线: 2 天
```

## 质量门检查清单

| # | 标准 | 证据来源 | 状态 |
|---|------|---------|------|
| 1 | 架构覆盖 100% 规范需求 | 高级项目经理任务列表与架构交叉引用 | ☐ |
| 2 | 品牌系统完整 (Logo、颜色、字体、调性) | 品牌守护者交付物 | ☐ |
| 3 | 所有技术组件有实施路径 | 后端架构师 + UX 架构师规范 | ☐ |
| 4 | 预算已批准且在约束范围内 | 财务追踪器计划 | ☐ |
| 5 | 冲刺计划基于速度且切合实际 | 冲刺优先排序器待办事项 | ☐ |
| 6 | 安全架构已定义 | 后端架构师安全规范 | ☐ |
| 7 | 合规要求已整合到架构中 | 法律要求映射到技术决策 | ☐ |

## 门决策

**双重签署要求**: 制作总监 (战略) + 现实检验员 (技术)

- **批准**: 进入阶段 2，附带完整架构包
- **修订**: 特定项目需要返工 (返回相关步骤)
- **重构**: 架构存在根本问题 (重新开始阶段 1)
